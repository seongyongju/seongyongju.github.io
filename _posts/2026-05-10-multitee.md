## Introduction

TEE(Trusted Execution Environment)는 일반적으로 하나의 device 내부에서 sensitive operation을 보호하는 실행 환경으로 사용됩니다. 예를 들어 key management, biometric authentication, secure payment, digital identity verification 같은 기능은 normal world의 OS가 compromise되더라도 TEE 내부에서 격리되어 실행될 수 있습니다. 그러나 실제 사용자 환경은 점점 단일 device 중심이 아니라 여러 device가 함께 동작하는 형태로 바뀌고 있습니다.

Smartphone, smartwatch, laptop, wearable, vehicle controller처럼 서로 다른 성능과 보안 기능을 가진 device들이 하나의 사용자 경험을 구성합니다. 이때 민감한 작업이 하나의 device 안에서 끝나지 않고 여러 device 사이에 걸쳐 수행된다면, single-device TEE만으로는 전체 trust boundary를 표현하기 어렵습니다. 한 device의 TEE가 안전하더라도, 다른 device와 통신하는 경계가 취약하면 전체 workflow는 안전하다고 보기 어렵습니다.

MultiTEE는 이러한 문제를 해결하기 위해 여러 heterogeneous device의 TEE를 연결하여 distributed TEE처럼 동작하게 하는 architecture를 제안합니다. 핵심 질문은 “서로 다른 device에 있는 TEE들이 어떻게 서로를 신뢰하고, 안전하게 데이터를 주고받으며, 하나의 trusted application처럼 협력할 수 있는가”입니다.

## Framework

### TEE-to-TEE Secure Channel

MultiTEE의 첫 번째 핵심은 TEE-to-TEE secure channel입니다. 일반적인 TLS 연결은 통신 상대가 특정 server certificate를 가지고 있다는 것은 검증할 수 있지만, 상대 device의 TEE 내부에서 어떤 trusted application이 실행 중인지까지 자동으로 보장하지는 않습니다. MultiTEE는 remote attestation을 이용해 각 device의 TEE 상태와 software stack integrity를 확인한 뒤, 그 결과에 기반하여 secure channel을 구성합니다.

이 과정에서 attestation은 단순 인증 이상의 의미를 가집니다. Device A의 TEE는 Device B가 실제 TEE 안에서 기대한 code를 실행하고 있는지 확인해야 하고, 반대로 Device B도 Device A의 상태를 검증해야 합니다. 이 상호 검증이 끝난 뒤에야 두 TEE 사이에서 sensitive data를 교환할 수 있습니다.

이 설계는 distributed setting에서 매우 중요합니다. 여러 device가 협력하는 순간 공격면은 각 device 내부뿐 아니라 device 간 communication path까지 확장됩니다. MultiTEE는 이 경계를 명시적으로 trust boundary 안에 포함시키려는 접근입니다.

### Distributed Trusted Application Interface

두 번째 핵심은 distributed Trusted Application interface입니다. 기존 TEE programming model에서는 normal world application이 TEE 내부의 Trusted Application을 호출하거나, TEE 내부 component가 local resource를 사용하는 방식이 일반적입니다. MultiTEE는 이 모델을 여러 device로 확장합니다.

즉 Trusted Application은 target service가 local TEE 안에 있는지, remote TEE 안에 있는지 가능한 한 투명하게 사용할 수 있어야 합니다. 이를 위해 MultiTEE는 remote procedure call과 유사한 abstraction을 제공하고, 호출 과정에서 필요한 attestation, secure channel setup, message protection을 architecture 차원에서 처리합니다.

이 abstraction이 중요한 이유는 application developer가 매번 device discovery, attestation protocol, key exchange, secure message format을 직접 구현하면 오류가 발생하기 쉽기 때문입니다. MultiTEE는 distributed trust를 application logic 아래의 system layer로 끌어내려, TEE 간 협력을 더 안전하고 재사용 가능한 형태로 만들려고 합니다.

## Experiments

저자들은 EUDI wallet use case를 일부 구현한 proof-of-concept를 통해 heterogeneous device cluster에서 MultiTEE가 동작할 수 있음을 보입니다. Digital identity wallet은 여러 device가 함께 참여하기 쉬운 대표적인 예시입니다. 예를 들어 credential storage는 한 device에 있고, user confirmation은 wearable에서 이루어지며, presentation은 smartphone이나 laptop에서 수행될 수 있습니다. 이런 상황에서는 각 device의 trusted component가 서로를 검증하고 협력해야 합니다.

평가에서 중요한 점은 MultiTEE가 단순히 이론적인 protocol 제안에 그치지 않고, 실제 TEE 환경과 device cluster에서 구현 가능성을 보인다는 것입니다. 또한 Rust 기반 memory-safe implementation을 사용하여 TEE 내부 구현 취약점을 줄이려는 설계를 적용합니다. TEE는 작은 trusted computing base를 목표로 하지만, TEE 내부 code에 memory corruption bug가 있으면 보호 경계 자체가 약해질 수 있습니다. 따라서 memory-safe language를 사용하는 것은 distributed TEE 환경에서도 중요한 engineering choice입니다.

논문은 performance 측면에서도 remote attestation과 secure channel setup이 추가 비용을 만들지만, 민감한 workflow에서 사용할 수 있는 수준의 overhead로 구성될 수 있음을 보입니다. 특히 매 호출마다 전체 attestation을 반복하는 것이 아니라, 검증된 TEE 사이의 secure session을 재사용하는 구조가 중요합니다.

## Conclusion

MultiTEE는 TEE를 단일 chip 내부 격리 기술로만 보지 않고, 여러 device가 협력하는 trust domain으로 확장합니다. 이 관점은 실제 시스템에서 중요합니다. 사용자의 민감한 작업은 점점 하나의 device 안에서 닫히지 않고, 여러 device와 controller, cloud service, peripheral이 함께 관여하는 형태가 됩니다. 이런 환경에서는 “각 device 안의 TEE가 안전하다”는 주장만으로는 부족하고, TEE 사이의 관계까지 안전하게 설계해야 합니다.

제가 이 논문에서 중요하게 본 부분은 distributed attestation입니다. SDV나 mobility 환경에서도 여러 ECU, domain controller, gateway, external service가 함께 동작합니다. 특정 controller 내부에 TEE를 넣는 것만으로는 충분하지 않고, 서로 다른 trusted component가 어떤 순서로 서로를 검증하고 어떤 channel로 데이터를 전달하는지까지 architecture에 포함되어야 합니다. MultiTEE는 이러한 문제를 일반적인 distributed TEE abstraction으로 정리했다는 점에서 참고할 만합니다.
