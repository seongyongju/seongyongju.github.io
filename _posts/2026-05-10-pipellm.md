## Introduction

Confidential computing 환경에서 LLM service를 제공하려면 모델 weight, user prompt, intermediate activation, output token 등 민감한 정보가 외부 cloud operator나 compromised system software로부터 보호되어야 합니다. 기존 confidential computing은 CPU TEE를 중심으로 발전했지만, LLM inference는 대부분 GPU에서 수행되기 때문에 GPU memory와 CPU-GPU data path까지 보호 범위에 포함해야 합니다.

NVIDIA H100과 같은 confidential GPU는 이러한 보호 기능을 제공할 수 있지만, 보안 기능은 공짜가 아닙니다. Confidential mode에서는 CPU와 GPU 사이의 data transfer, GPU memory 접근, encryption/decryption 과정이 추가되면서 serving throughput이 감소할 수 있습니다. LLM service는 latency와 throughput이 비용에 직접 연결되기 때문에, 보안 기능이 성능을 크게 떨어뜨리면 실제 deployment가 어려워집니다.

PipeLLM은 confidential GPU 환경에서 LLM service의 성능 overhead를 줄이기 위한 runtime system입니다. 논문의 핵심 문제의식은 “암호화를 제거하지 않고, 암호화 비용이 critical path에 드러나는 방식을 줄일 수 있는가”입니다. 이를 위해 PipeLLM은 LLM serving의 반복적이고 예측 가능한 실행 패턴을 활용하여 encryption과 GPU computation을 pipeline으로 겹쳐 실행합니다.

## Framework

### Speculative Pipelined Encryption

이 논문의 핵심 아이디어는 encryption과 GPU computation을 pipeline 방식으로 겹쳐 실행하는 것입니다. 일반적인 방식에서는 GPU가 필요한 데이터를 요청한 뒤 암호화 또는 전송 준비가 수행되기 때문에, GPU가 data를 기다리는 구간이 생깁니다. 이 구간은 전체 serving latency와 throughput에 직접적인 영향을 줍니다.

PipeLLM은 LLM inference가 token-by-token으로 진행되고, 각 단계에서 필요한 data movement pattern이 어느 정도 예측 가능하다는 점을 활용합니다. Runtime은 앞으로 필요한 data를 미리 예상하고, GPU가 현재 computation을 수행하는 동안 다음 단계에 필요한 encryption work를 시작합니다. 이렇게 하면 encryption cost가 computation과 겹쳐져서 사용자가 관찰하는 critical path에서는 작게 드러납니다.

여기서 speculative이라는 표현이 중요한 이유는 미래에 필요한 data를 완전히 확정적으로 알 수 없기 때문입니다. Token generation은 model output에 따라 다음 step이 결정되지만, serving system은 workload structure와 batching pattern을 이용해 높은 확률로 필요한 data를 미리 준비할 수 있습니다. 잘못 예측한 경우에는 추가 비용이 생길 수 있지만, 전체적으로는 encryption latency를 숨기는 효과가 큽니다.

### LLM Serving Pattern 활용

LLM inference는 크게 prompt processing(prefill)과 token generation(decode)으로 나눌 수 있습니다. Prefill 단계에서는 입력 prompt 전체를 처리하며 많은 parallel computation이 발생하고, decode 단계에서는 각 request가 한 token씩 생성됩니다. Decode 단계에서는 KV-cache를 반복적으로 읽고 갱신하기 때문에 memory traffic과 scheduling이 성능에 큰 영향을 줍니다.

PipeLLM은 이러한 serving pattern을 runtime optimization의 근거로 사용합니다. 단순히 GPU confidential computing을 켜고 성능 저하를 받아들이는 것이 아니라, LLM workload가 어떤 순서로 data를 요구하는지 관찰하여 encryption과 transfer를 재배치합니다. 즉 보안 기능의 비용을 workload-aware scheduling으로 흡수하려는 접근입니다.

이 관점은 중요한 engineering insight를 줍니다. 보안 기능을 추가할 때 overhead가 생기는 것은 피하기 어렵지만, 그 overhead가 반드시 application의 critical path에 그대로 나타날 필요는 없습니다. Runtime이 workload 특성을 잘 이용하면 보안 기능을 유지하면서도 성능 저하를 줄일 수 있습니다.

## Experiments

논문은 confidential GPU에서 암호화가 LLM serving 성능에 미치는 영향을 분석하고, PipeLLM이 이 overhead를 줄일 수 있음을 보입니다. 평가의 핵심은 baseline confidential serving에서 어떤 지점이 bottleneck이 되는지 확인하고, pipelined encryption이 그 bottleneck을 얼마나 줄이는지 측정하는 것입니다.

LLM serving 성능은 단일 요청 latency뿐 아니라 batch 처리량, token generation throughput, memory utilization에 의해 결정됩니다. PipeLLM은 encryption 작업을 computation과 겹치게 만들어 GPU idle time을 줄이고, confidential mode에서 발생하는 throughput 저하를 완화합니다. 특히 많은 사용자가 동시에 요청하는 online serving scenario에서는 작은 per-token overhead도 누적되어 큰 비용이 되므로, pipeline을 통한 latency hiding이 중요합니다.

이 논문의 결과를 볼 때 핵심은 보안 기능을 제거하지 않았다는 점입니다. Confidential computing의 보호 속성을 유지하면서 runtime scheduling과 data preparation 방식을 바꿔 성능을 개선합니다. 이는 security-performance trade-off를 단순히 “보안을 낮추면 빨라진다”로 보지 않고, architecture와 runtime design으로 개선할 수 있음을 보여줍니다.

## Conclusion

제가 이 논문에서 중요하게 본 부분은 보안 기능을 제거하지 않고도 overhead를 줄이는 설계 방식입니다. Confidential computing의 비용을 단순히 감수하는 것이 아니라, workload pattern을 이용해 보호 기능의 지연을 숨기는 접근은 SDV controller architecture에서 TEE overhead를 줄이는 문제와도 연결됩니다.

Controller architecture에 TEE나 TrustZone을 도입할 때도 암호화, world switch, protected memory access, attestation 같은 비용이 생깁니다. 이 비용을 단순히 줄이기 어렵다면, 어떤 작업과 겹쳐 실행할 수 있는지, 어떤 data path를 미리 준비할 수 있는지, 어떤 보안 연산이 critical path에 놓이는지 분석해야 합니다. PipeLLM은 LLM serving이라는 다른 domain의 논문이지만, security overhead를 workload-aware runtime design으로 줄인다는 점에서 hardware security optimization 관점에 직접적인 참고가 됩니다.
