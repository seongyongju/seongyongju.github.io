## Introduction

기존 OS는 보통 hardware address space를 이용해 subsystem을 격리합니다. Process boundary, page table, privilege level 같은 하드웨어 기반 메커니즘은 강한 isolation을 제공하지만, isolation domain을 자주 넘나들어야 하는 kernel 내부에서는 비용이 큽니다. Context switch, TLB flush, data copy 같은 비용 때문에 fine-grained isolation을 넓게 적용하기 어렵고, 그 결과 많은 kernel subsystem이 같은 address space 안에서 실행됩니다.

문제는 kernel 내부 code가 모두 같은 신뢰 수준을 갖는 것은 아니라는 점입니다. Device driver, file system, network stack처럼 복잡하고 외부 입력을 많이 처리하는 component는 bug가 발생하기 쉽습니다. 이 component들이 같은 kernel address space를 공유하면 하나의 memory safety bug가 전체 kernel compromise로 이어질 수 있습니다.

RedLeaf는 Rust의 type safety와 memory safety를 활용해 hardware address space 없이도 kernel subsystem을 격리할 수 있는 safe operating system design을 제안합니다. 핵심은 isolation을 hardware boundary가 아니라 language boundary로 일부 이동시키는 것입니다. 이를 통해 kernel 내부 component를 더 작은 단위로 나누면서도, hardware isolation이 만드는 큰 성능 비용을 피하려고 합니다.

## Framework

### Language-based Isolation Domain

RedLeaf는 lightweight language-based isolation domain을 도입합니다. 각 domain은 information hiding과 fault isolation의 단위가 되며, 동적으로 load되고 cleanly terminate될 수 있습니다. 중요한 점은 domain이 서로 같은 address space를 공유하더라도, Rust type system과 compiler-enforced rule을 통해 임의의 memory access를 막는다는 것입니다.

Hardware isolation에서는 component가 다른 address space에 있으면 pointer를 직접 공유하기 어렵고, data를 복사하거나 IPC를 거쳐야 합니다. RedLeaf는 language-level ownership과 borrowing rule을 이용해 domain 사이의 object transfer를 안전하게 표현합니다. 어떤 object가 어느 domain에 속하는지, 어느 시점에 누가 접근 권한을 갖는지 type system이 추적하기 때문에, use-after-free나 dangling pointer 같은 오류를 구조적으로 줄일 수 있습니다.

또한 RedLeaf는 unsafe code를 완전히 제거한다고 주장하기보다는, unsafe code가 필요한 부분을 작게 제한하고 그 위에 safe abstraction을 제공합니다. 시스템 programming에서는 hardware register 접근, DMA, 외부 device와의 interaction 때문에 unsafe가 필요한 경우가 있습니다. RedLeaf의 목표는 unsafe를 통제 가능한 작은 trusted computing base로 제한하는 것입니다.

### Zero-copy Communication

Rust의 ownership과 type system을 활용하면 domain 사이에서 zero-copy communication을 설계할 수 있습니다. 예를 들어 network packet buffer를 driver domain에서 network stack domain으로 전달할 때, 실제 data를 복사하는 대신 buffer ownership을 이동시킬 수 있습니다. Ownership이 이동하면 이전 domain은 더 이상 그 buffer에 접근할 수 없고, 새 domain이 독점적으로 사용할 수 있습니다.

이 방식은 isolation과 performance 사이의 trade-off를 다르게 만듭니다. Hardware isolation에서는 안전하게 데이터를 전달하기 위해 copy가 필요한 경우가 많지만, RedLeaf는 type-checked ownership transfer로 같은 address space 안에서도 안전한 zero-copy 전달을 가능하게 합니다. 따라서 고성능 I/O path에서 isolation을 포기하지 않고도 overhead를 줄일 수 있습니다.

RedLeaf의 communication model은 단순한 function call보다 엄격합니다. Domain boundary를 넘는 interface는 명시적으로 정의되고, 전달 가능한 object type과 lifetime이 제한됩니다. 이를 통해 domain 내부 구현은 숨기면서도 필요한 기능만 노출할 수 있습니다.

## Experiments

논문은 POSIX-subset OS인 Rv6를 RedLeaf domain들의 조합으로 구현하고, network와 NVMe driver 성능을 평가합니다. 평가의 핵심은 language-based isolation이 실제 kernel architecture에서 사용할 수 있을 만큼 충분한 성능을 내는지 확인하는 것입니다.

특히 network와 storage driver는 data path가 길고 packet 또는 block I/O가 빠르게 처리되어야 하므로, isolation overhead가 조금만 커져도 전체 throughput에 영향을 줍니다. RedLeaf는 zero-copy communication과 lightweight domain boundary를 이용해 이러한 I/O path에서도 fine-grained isolation을 유지할 수 있음을 보입니다.

논문은 RedLeaf가 DPDK, SPDK 같은 고성능 user-space I/O framework와 비교 가능한 수준의 driver 구현 가능성을 보인다고 설명합니다. 여기서 중요한 점은 단순히 성능만 맞추는 것이 아니라, unsafe한 driver code가 전체 kernel을 망가뜨리지 않도록 isolation domain 안에 가두면서도 성능을 유지한다는 것입니다.

## Conclusion

RedLeaf는 hardware isolation만이 OS security의 유일한 방법은 아니라는 점을 보여줍니다. Memory-safe language를 system design의 중심에 두면 isolation과 performance 사이의 trade-off를 다르게 설계할 수 있습니다. 특히 OS 내부처럼 성능 요구가 강하고 component 간 상호작용이 빈번한 환경에서는 language-based isolation이 의미 있는 대안이 될 수 있습니다.

제가 이 논문에서 중요하게 본 부분은 보안 경계를 항상 hardware boundary로만 생각하지 않는다는 점입니다. TEE나 TrustZone도 hardware-assisted isolation이지만, 모든 boundary를 하드웨어로 나누면 context switch와 communication overhead가 커질 수 있습니다. RedLeaf는 type system, ownership, interface design을 활용해 더 가벼운 isolation을 만드는 방법을 보여줍니다. 이는 controller architecture에 보안 기능을 넣을 때도, 어떤 부분은 hardware isolation으로 보호하고 어떤 부분은 language/runtime abstraction으로 보호할지 나누어 생각하게 합니다.
