## Introduction

Rust 같은 memory-safe language는 많은 시스템 취약점을 구조적으로 줄일 수 있습니다. Ownership, borrowing, lifetime checking을 통해 use-after-free, double free, data race 같은 버그를 컴파일 단계에서 줄일 수 있기 때문입니다. 하지만 실제 시스템은 Rust만으로 완전히 작성되지 않습니다. 기존 C/C++ library, device driver, OS component, vendor SDK와 상호작용해야 하는 경우가 많습니다.

이때 FFI(Foreign Function Interface) 경계가 safe language의 soundness를 다시 깨뜨릴 수 있습니다. Rust code가 아무리 안전해도, foreign library가 잘못된 pointer를 저장하거나, Rust object의 lifetime을 무시하거나, memory를 임의로 수정하면 전체 프로그램의 safety guarantee가 약해집니다.

Omniglot은 unmodified and untrusted foreign library와 상호작용하면서도 Rust의 safety와 soundness를 유지하기 위한 framework입니다. 핵심은 foreign code를 완전히 신뢰하지 않고, Rust와 foreign library 사이의 interaction을 안전하게 제한하는 것입니다.

## Framework

### Foreign Library Isolation

Omniglot은 foreign code를 memory isolation primitive 안에 배치하고, Rust와 foreign library 사이의 interaction을 validation infrastructure로 관리합니다. Foreign library는 Rust object에 직접 임의 접근할 수 없고, 정해진 interface를 통해서만 데이터를 주고받습니다. 이를 통해 foreign library가 Rust object의 memory safety를 깨뜨리지 못하도록 합니다.

이 방식은 language safety와 hardware/software isolation을 결합합니다. Rust 내부에서는 type system이 safety를 보장하고, Rust 바깥의 foreign code는 isolation boundary 안에 가둡니다. 중요한 점은 foreign library를 수정하지 않는다는 것입니다. 실제 시스템에서는 기존 library를 전부 Rust로 다시 쓰기 어렵기 때문에, unmodified library와 안전하게 상호작용하는 방법이 필요합니다.

### Efficient Interaction

단순히 모든 데이터를 copy하거나 serialize하면 안전성은 얻을 수 있지만 overhead가 큽니다. System software에서는 library call이 자주 발생하고 data structure가 크기 때문에, 매번 deep copy를 수행하면 성능이 크게 떨어질 수 있습니다.

Omniglot은 안전한 interaction을 제공하면서도 불필요한 copy와 serialization을 피하는 방향으로 설계됩니다. 이를 위해 어떤 object를 foreign code에 노출할 수 있는지, foreign code가 어떤 방식으로 object를 참조할 수 있는지, call이 끝난 뒤 ownership과 validity가 어떻게 유지되는지 검증합니다. 즉 safety를 얻기 위해 모든 상호작용을 가장 보수적인 방식으로 막는 것이 아니라, 필요한 interaction은 효율적으로 허용합니다.

## Experiments

논문은 constrained embedded OS kernel과 Linux userspace library integration이라는 두 가지 환경에서 Omniglot을 구현하고 평가합니다. Embedded OS kernel에서는 resource가 제한적이고 low-level interaction이 많기 때문에, FFI boundary의 overhead가 특히 중요합니다. Linux userspace library integration은 기존 ecosystem과 safe language runtime이 어떻게 공존할 수 있는지 보여줍니다.

결과적으로 Omniglot은 더 약한 guarantee를 제공하는 방식과 comparable한 성능을 보이면서, 유사한 safety guarantee를 제공하는 기존 방식보다 효율적임을 보입니다. 이는 foreign code와의 상호작용에서 safety와 performance를 동시에 고려한 설계가 가능하다는 점을 보여줍니다.

## Conclusion

Omniglot은 safe language 기반 system에서도 foreign code boundary가 핵심 공격면이 될 수 있음을 보여줍니다. 시스템 보안을 설계할 때 language safety만 보는 것이 아니라, unsafe component와의 경계까지 함께 설계해야 합니다.

제가 이 논문에서 중요하게 본 부분은 “safe language를 쓰면 끝”이 아니라는 점입니다. 실제 controller나 OS architecture에서는 legacy C library, hardware driver, vendor binary와의 상호작용이 피할 수 없습니다. 따라서 보안 경계는 language 내부뿐 아니라 foreign interface에도 있어야 합니다. 이는 TEE나 TrustZone을 넣을 때 trusted component와 untrusted legacy component가 어떻게 상호작용해야 하는지 고민하는 데도 연결됩니다.
