## Introduction

Modern vehicle은 IVN(In-Vehicle Network)과 EVN(Extra-Vehicle Network)을 연결하여 navigation, diagnostics, OTA update, cloud-assisted service 같은 기능을 제공합니다. 과거 차량 네트워크는 주로 차량 내부 ECU들 사이의 통신을 다루었지만, connected vehicle에서는 외부 네트워크에서 들어오는 정보가 차량 내부 제어 흐름에 영향을 줄 수 있습니다. 이때 EVN과 IVN 사이에 있는 gateway는 외부 입력이 차량 내부 안전-critical component로 전달되는 것을 통제하는 중요한 위치에 있습니다.

기존 gateway 설계는 내부 ECU와 gateway를 비교적 신뢰된 구성요소로 보고, message filtering이나 policy enforcement를 수행하는 방식이 많았습니다. 하지만 EVN platform은 일반 OS, 복잡한 application, network stack, external interface를 포함하기 때문에 compromise 가능성이 큽니다. 공격자가 EVN manager 또는 EVN application을 장악하면, IVN gateway로 전달되는 control message를 위조하거나 조작할 수 있습니다. 이 경우 gateway가 message의 출처와 무결성을 신뢰할 수 없다면, 외부 침해가 차량 내부 제어 네트워크로 확산될 수 있습니다.

SECV는 이러한 상황에서 EVN manager가 compromise되더라도 IVN gateway가 EVN-originated control message를 신뢰성 있게 검증할 수 있도록 하는 runtime security mechanism입니다. 핵심은 EVN 쪽에서는 TEE를 이용해 message generation과 validation을 보호하고, IVN gateway 쪽에서는 HSM을 이용해 cryptographic proof를 검증하는 것입니다. 즉 복잡한 EVN software stack 전체를 신뢰하지 않고, hardware trust anchor가 보호하는 작은 security-critical path만 신뢰하도록 구조를 바꿉니다.

## Framework

### TEE-based Mediation

SECV는 EVN-to-IVN traffic을 TEE 내부에서 중재합니다. EVN application이 IVN으로 message를 보내려면, 해당 message는 TEE 내부의 trusted component를 거쳐야 합니다. 이 trusted component는 application별 validation rule을 적용하고, message가 허용된 형식과 policy를 만족하는지 확인합니다. 이렇게 하면 normal world의 EVN OS나 application이 compromise되더라도, 공격자가 임의의 control message를 그대로 IVN에 주입하기 어려워집니다.

TEE 내부 validation의 역할은 단순 signature 생성이 아닙니다. 어떤 application이 어떤 종류의 IVN message를 만들 수 있는지, message field가 허용 범위 안에 있는지, runtime context가 적절한지 등을 확인하는 mediation layer로 동작합니다. 이후 TEE는 검증된 message에 cryptographic proof를 붙입니다. Gateway는 이 proof를 통해 message가 TEE 내부의 trusted path를 거쳤는지, 전송 중 조작되지 않았는지 확인할 수 있습니다.

이 구조의 장점은 EVN platform 전체를 trusted computing base에 넣지 않는다는 점입니다. Linux나 Android 기반 EVN stack 전체를 신뢰하려고 하면 공격면이 너무 커집니다. SECV는 TEE 내부의 작은 validation logic과 key material만 신뢰하도록 하여, compromise된 EVN manager가 있더라도 gateway가 신뢰 가능한 판단을 내릴 수 있게 합니다.

### HSM-based Verification

IVN gateway는 HSM(Hardware Security Module)을 이용해 proof를 검증합니다. HSM은 차량 환경에서 cryptographic key와 security-critical operation을 보호하기 위해 사용되는 hardware trust anchor입니다. Gateway가 normal software만으로 proof를 검증하면, gateway software compromise나 key leakage가 발생했을 때 protection이 약해질 수 있습니다. HSM을 사용하면 key material을 hardware boundary 안에 보호하고, verification operation도 더 신뢰성 있게 수행할 수 있습니다.

TEE와 HSM을 함께 사용하는 구조는 역할 분리가 명확합니다. EVN 쪽 TEE는 message가 trusted validation path를 통과했음을 보장하고, IVN gateway의 HSM은 그 보증을 검증합니다. 즉 TEE는 sender-side trust anchor이고, HSM은 receiver-side trust anchor입니다. 둘 사이에 cryptographic binding이 존재하기 때문에, EVN OS가 위조 message를 만들거나 기존 message를 변조하더라도 gateway는 이를 걸러낼 수 있습니다.

이 설계는 connected vehicle에서 trust boundary를 다시 정의합니다. 기존에는 gateway가 EVN에서 온 message를 software policy로 처리했다면, SECV에서는 message 자체가 hardware-backed proof를 포함해야 합니다. 따라서 gateway는 “어느 software process가 보냈는가”보다 “hardware trust anchor가 검증한 message인가”를 기준으로 판단합니다.

## Experiments

논문은 ARM TrustZone과 EVITA-compliant HSM을 갖춘 automotive SoC에서 SECV를 구현하고 평가합니다. 평가의 핵심은 보안 기능을 추가했을 때 EVN-to-IVN communication이 차량의 real-time constraint를 만족할 수 있는지 확인하는 것입니다. 차량 네트워크에서는 message delay가 크거나 loss가 증가하면 제어 안정성과 사용자 경험에 영향을 줄 수 있기 때문에, security mechanism은 lightweight해야 합니다.

SECV의 overhead는 크게 TEE 내부 validation 비용, cryptographic proof 생성 비용, gateway HSM verification 비용, message size 증가로 나눌 수 있습니다. 논문은 이러한 비용이 실제 automotive hardware에서 감당 가능한 수준인지 측정합니다. 결과적으로 SECV는 낮은 전송 overhead와 제한적인 message loss 증가를 보이며, EVN-originated attack을 완화할 수 있음을 보입니다.

이 실험에서 중요한 점은 단순히 cryptographic scheme이 빠르다는 것을 보이는 것이 아니라, TEE와 HSM이라는 실제 차량용 hardware trust anchor를 사용했을 때 end-to-end path가 동작한다는 것입니다. SDV 환경에서는 security mechanism이 설계상 안전해도 실제 controller architecture에 넣었을 때 latency나 throughput 문제로 사용하기 어려울 수 있습니다. SECV는 이 practical constraint를 함께 다룹니다.

## Conclusion

SECV는 connected vehicle에서 hardware trust anchor가 controller architecture 보안에 어떻게 들어갈 수 있는지 보여주는 사례입니다. 특히 TEE와 HSM의 역할을 나누어, 복잡한 EVN software stack이 compromise되더라도 IVN gateway가 신뢰 가능한 판단을 할 수 있도록 설계한 점이 중요합니다.

제가 이 논문에서 중요하게 본 부분은 보안 경계를 “전체 EVN을 믿는다”에서 “TEE가 보증한 message만 믿는다”로 바꾼 점입니다. SDV 환경에서는 controller와 service가 복잡해질수록 모든 software를 동일하게 신뢰하는 방식은 유지되기 어렵습니다. 따라서 message path 중 어느 부분을 hardware-backed trust anchor로 감싸고, gateway가 어떤 proof를 검증해야 하는지 설계하는 것이 핵심입니다. SECV는 이러한 security architecture를 vehicle network context에서 구체적으로 보여줍니다.
